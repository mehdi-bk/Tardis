## Step 1 — Data Cleaning & Exploration (`tardis_eda.ipynb`)

### What is done

**Loading the data**
```python
df = pd.read_csv("project_dataset.csv", sep=";", decimal=",")
```
The source file uses `;` separators and `,` decimals (European format).

**Dropping useless columns**
Comment columns (11,000+ missing values out of 12,000 rows) are removed as they provide no analytical value.

**Removing duplicates**
174 duplicate rows are identified and removed.

**Text standardization**
All text columns are uppercased and stripped of extra spaces to avoid duplicates caused by casing (`"paris"` ≠ `"PARIS"`).

**Station name correction**
Inconsistent abbreviations are normalized (`"ST MALO"` → `"SAINT MALO"`, etc.).

**Data type conversion**
Numeric columns stored as text are converted using `pd.to_numeric()`. Dates are parsed with a custom function handling multiple formats (`"2018-01"`, `"01-2018"`, etc.).

**Handling missing values**
Numeric columns are filled with the median. Rows missing a departure station, arrival station, or date are dropped.

**Feature Engineering**
- `year` : year extracted from the date
- `month` : month extracted from the date
- `cancellation_rate` : cancellation rate = cancelled trains / scheduled trains

**Saving**
```python
df.to_csv("cleaned_dataset.csv", index=False)
```

### Visualizations produced

- Arrival delay distribution (histogram + KDE)
- Top 10 most delayed departure stations (horizontal bar chart)
- Average delay trend by year (line chart)
- Breakdown of delay causes (pie chart)

## Step 2 — Modeling (`tardis_model.ipynb`)

### Full Pipeline

**1. Outlier removal**  
Extreme values (delays at -472 min or +92 min) are removed by cutting the bottom and top 1% of the target distribution. This step improved R² from 0.06 to 0.81.

**2. Fixing missing and infinite values**  
Infinite values caused by division by zero (e.g., `cancellation_rate`) are replaced with `NaN` and then filled with the median.

**3. Defining X and y**
- **y** (target) : `Retard moyen de tous les trains à l'arrivée` (average arrival delay)
- **X** (features) : 16 columns including stations, service type, duration, delay causes, year, month, cancellation rate

**4. Encoding categorical variables**  
Two strategies used depending on the model:
- `LabelEncoder` for Linear Regression (saved as `.joblib`)
- `pd.get_dummies()` for Random Forest → binary columns per category

**5. Train/test split**  
80% training / 20% testing with `random_state=42` for reproducibility.

**6. Scaling (Linear Regression only)**  
`StandardScaler` applied to put all features on the same scale (mean=0, std=1). Fitted only on training data to avoid data leakage.

### Models trained

| Model | RMSE | MAE | R² |
|---|---|---|---|
| Baseline (mean) | 3.51 min | — | — |
| Linear Regression | 3.10 min | 2.34 min | 0.2231 |
| **Random Forest** | **1.52 min** | **1.05 min** | **0.8122** |

**Random Forest parameters:**
```python
RandomForestRegressor(
    n_estimators=200,   # 200 decision trees
    random_state=42,    # reproducibility
    n_jobs=-1,          # use all CPU cores
    min_samples_leaf=2  # prevents overfitting
)
```

**Saving the model:**
```python
joblib.dump(rf, "model.joblib")
joblib.dump(X_rf.columns.tolist(), "model_features.joblib")
```

### How prediction works

The user selects a departure station, arrival station, service type, year and month. The model then:

1. Looks up the **historical averages for that specific route** (duration, cancellation rate, delay causes, etc.)
2. Falls back to **global dataset averages** if the route has never been seen before
3. One-hot encodes the categorical inputs to match the training format
4. Aligns the columns exactly with `model_features.joblib`

```python
# One-hot encode categorical inputs
service_col = f"Service_{service}"
dep_col = f"Gare de départ_{dep}"
arr_col = f"Gare d'arrivée_{arr}"

# Set matching columns to 1
input_row[service_col] = 1
input_row[dep_col] = 1
input_row[arr_col] = 1

prediction = float(model.predict(input_row)[0])
```

### Note on year range

The prediction form allows years from **2024 to 2030**, which goes beyond the training data (2018–2024). The model does not forecast future trends — it extrapolates based on historical patterns learned from past data. Predictions for future years are estimates assuming that delay patterns remain similar to what was observed historically. This limitation should be kept in mind when interpreting results.
